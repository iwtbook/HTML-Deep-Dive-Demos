Article

General Syntax & Usage

According to the current HTML Standard, the <article> element represents a complete, or self-contained, composition in a document, page, application, or site. The examples they give are forum posts, magazines, newspaper articles, or blog entries. While most of these are long forms of content, it is important to note that length is not a requirement of the <article> element, only that its content is self contained. As such, there could be one <article> element for an entire NYT online article, and one <article> element for each individual comment on that same article. Below are some examples of the <article> element being used in context:

For a long article about a recipe

<article>
  <header>
    <h1>How to make Chocolate Chip Cookies</h1>
  </header>
  <main>
    <section id="ingredients">
      <ul>
        <li>Chocolate Chips</li>
        <li>Eggs</li>
        <li>Flour</li>
        <li>Flour</li>
        <li>...</li>
      </ul>
    </section>
    <section id="instructions">
      <section id="instructionsWet">
        <p>First, add your flour to your eggs...</p>
      </section>
      <section id="instructionsDry">
        <p>...</p>
      </section>
    <section>
  </main>
</article>
 
For a short, single, self-contained comment

<article class="user-comment">
  <img src="/profiles/CamdynR.png" alt="CamdynR's profile photo">
  <p class="username">CamdynR</p>
  <p class="comment>First</p>
</article>

<article> elements are also allowed to nest other articles. One common use for nesting them is for when you have comments on an article or post, as was said above. Although the comments are self-contained, they are still relevant to the main article or post, so they usually get their own <article> element but nested inside of the main <article> element for the article. Below is the same two examples as above, but combined to show how they might fit together.

For a long article about a recipe
      
<article>
  <header>
    <h1>How to make Chocolate Chip Cookies</h1>
  </header>
  <main>
    <section id="ingredients">
      <ul>
        <li>Chocolate Chips</li>
        <li>Eggs</li>
        <li>Flour</li>
        <li>Flour</li>
        <li>...</li>
      </ul>
    </section>
    <section id="instructions">
      <section id="instructionsWet">
        <p>First, add your flour to your eggs...</p>
      </section>
      <section id="instructionsDry">
        <p>...</p>
      </section>
    <section>
  </main>
  <section id="commentSection">
    <article class="userComment">
      <img src="/profiles/CamdynR.png" alt="CamdynR's profile photo">
      <p class="username">CamdynR</p>
      <p class="comment">First</p>
    </article>
  </section>
</article>

Properties

While the <article> element doesn't have any special attributes outside of the regular global ones, its semantic value brings up an interesting point when thinking about custom elements.

There are two types of custom elements:

Autonomous custom elements
These are more widely adopted and recognized, they are custome elements that don't inherit from any standard HTML elements. You use them by writing them out as any other element, but with a custom name that contains a hyphen, e.g. <custom-element>
Customized built-in elements
These custom elements aren't as widely adopted, but have a lot more potential. They build on top of a standard HTML element, in doing so inhereting its properties. To use it, you take the same name that you would use to creat an autonomous custom element, but you add it to a standard HTML element's is attribute. Here is an example: <h1 is="custom-element">

Customized built-in elements are powerful since you can then leverage an elements inherent semantic value, like <article>, while also adding extra function / styles to it. The more ubiquitous autonomous custom elements have no inherent meaning, so without JavaScript to create them, they lose all function and purpose. Without JavaScript, the <article> element will still act like a regular <article> element, so your code becomes less brittle.The only thing to watch out for is its browser adoption, since it is not compatable (as of July 28, 2020) with either IE 11 or Safari 13.1.

Accessibility

For most regular users the <article> element does not provide any extra visual enhancement. Using a <div> instead would hardly affect the page to abled users. The real benefits of the <article> element, or any semantic element for that matter, is the enhanced experience for users with screen readers. Using semantic elements helps the screen reading robot decipher what the content is so that it can better explain the page to its user. Because of the added benefits to visually impaired users, and a desire to keep the internet accessible to all, search engines have also started to prioritize webpages that use this semantic markup. So not only is your webpage more accessible for using them, but you get an SEO increase as well, all for not much more effort than using a <div>.

Misuse

The main confusion around the <article> element tends to come from getting it confused with the <section> element as they are semantically pretty similar. Their key difference though stems from the fact that the <article> element represents a complete, contained composition while a <section> element represents one complete part of a larger composition. From this you might assume that a <section> element must be nested within an <article> element, but this is not necessarily the case. <section> elements can divide any larger container, and <article> elements can be on their own without any subdivisions.Though generally it is useful to use the <section>
