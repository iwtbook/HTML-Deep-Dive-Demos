hr

General Use & Syntax

According to the current HTML Standard, the <hr> element represents a paragraph-level thematic break, e.g. a scene change in a story, or a transition to another topic within a section of a reference book. While it used to be known as the Horizontal Rule element (thus why it is <hr>), it is now referred to as the Thematic Break element to dissuade developers from using it for stylistic purposes (more on this in the Misuse section below). Below is an example of one being used in context:

<h1>Types of Rocks</h1>
<p>In this section we will be covering different kinds of rocks and how they're formed.</p>
<hr>
<h2>Igneous rocks</h2>
<p>...</p>

Misuse

As was stated above, the <hr> element used to be referred to as the "Horizontal Rule" element in previous versions of HTML. Due to this naming, the element was often used for purely stylistic purposes without any semantic meaning. With this element, the <font> element, and a few others, the line blurred between what technology was used for styling and what technology was used for markup. When HTML5 was being developed, it was decided that a clean line needed to be formed, so the <hr> element was given a completely semantic meaning of as a "Thematic Break", distancing itself from its stylistic past.

Since the <hr> element still renders as think horizontal gray line on the screen, it is still common for developers to confuse it for a styling tool. If you need a way to create a thin horizontal line across the screen without any semantic meaning, below is an example of one way it could be achieved using CSS:

HTML:
<div class="horizontalRule"></div>

CSS:
.horizontalRule {
  background-color: lightgray;
  height: 2px;
  margin: 0 auto;
  width: 90vw;
}

Trivia

Similar to the <br> element, the <hr> element is an empty element, meaning that since there's no reason to have content inside of it, it doesn't take a closing tag. Often times you might see <br> and <hr> written instead as <br/> or <hr/>. The reason for this goes all the way back to XML and XHTML, where these elements were apart of a group of elements referred to as "self-closing tags". The reason being that if you were unfamiliar with which elements were empty elements, you could at a glance determine which were which. This might prevent some slight worry if you didn't see a closing tag for an element. In XML and XHTML these self-closing tags were required to have that trailing forward slash, so a lot of developers got into the habit of always including them, even after XML and XHTML faded away. Nowadays those two markup languages aren't nearly as ubiquitous, but the tradition of including them has stuck around as a remnant of those days (even though they are no longer necessary).
